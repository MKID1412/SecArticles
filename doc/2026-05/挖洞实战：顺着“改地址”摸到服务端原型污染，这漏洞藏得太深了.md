有些网站后端在处理用户提交的   
JSON 数据时，会不小心让攻击者  
污染全局对象原型，进而篡改应用行为。今天咱们就手把手走一遍“地址变更”功能里的  
服务端原型污染实战，全程用 Burp 操作。  
  
第1步：正常改一次地址，让流量进历史记录  
  
登录目标账户，进入地址管理页面。  
  
填写表单，提交一次正常的账单地址或收货地址变更。  
  
不用急着抓包，正常走完流程就行。  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/qg1MKHx3jGE4EiaanJWLZqf4ksf9SENibmf6mDQmgKzsV5E16pE8hFb7XT1mvZXyyegC7iaavyp0F8qm1ODIKBDKW1PACGLZWrscHWR4hlwYB4/640?wx_fmt=png&from=appmsg "")  
  
第2步：在 Burp 里定位目标请求  
1. 切到  
burp工具的   
Proxy →   
HTTP history。  
  
1. 找到那条   
POST /my-account/change-address 请求。  
  
点开看一眼：请求体是 JSON 格式的地址字段，返回包里也直接回显了更新后的用户信息对象。  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/qg1MKHx3jGHIhyIZGj23Fn5pXdBNicibhiaA1ictjbTuyA0kOZcy06qZjFricv4wWNQgQFsKciamABSicbDP7J4UgVFpJhwup9c8asbiaycdJJt6uFU/640?wx_fmt=png&from=appmsg "")  
  
右键该请求，选择   
Send to Repeater。  
  
第3步：初探原型注入（可能没反应，别慌）  
  
在   
Repeater 的请求体中，往   
JSON 里加一个   
__proto__ 属性：  
```
"__proto__": {
    "foo": "bar"
}
```  
  
发送。  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/qg1MKHx3jGExtJtG5Xv8aUYWw95SaHicOBY7ibqZ1Hl9brVIzkE8oKRZiaALe1NO3oFIzCMdhLaDa1bIRcq19rYPNChgSBfgceED0763nha98M/640?wx_fmt=png&from=appmsg "")  
  
观察响应：返回的用户对象里看不到 foo 这个属性，这很正常，但不代表没漏洞。  
  
第4步：故意破坏 JSON，探探错误信息  
  
在请求体里故意搞个语法错误，比如删掉某行末尾的逗号。  
  
发送。  
  
看响应：虽然   
HTTP 状态码是 500，但返回的 JSON 错误对象里应该能找到一个   
status 字段，值是 400。  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/qg1MKHx3jGEQfkbUGIjybkYDBqQdQtVVIISs3z3nGvpZvobbrMLvKazH3O5gCMYsvz9icC2bTvXibnOzz9ia2ibVxR7OB5JaRmRiaVrXEnW3lqYU/640?wx_fmt=png&from=appmsg "")  
  
记住这个字段名“  
status”，它就是咱们下一步要攻击的目标。  
  
第5步：用自定义   
status   
值去污染  
服务端原型  
  
先把刚才故意搞出的语法错误修好，恢复合法   
JSON。  
  
修改注入内容，把目标对准那个   
status 属性，值设在 400–599 之间，比如：  
```
"__proto__": {
    "status": 500
}
```  
  
发送，确认能正常收到包含用户对象的响应。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/qg1MKHx3jGGiapibhRgoWwEM6icAhibMQZbqibdHSDBC2yD92LmFbpOelicXpunthn1akuHrcrvfQBjYYkH2zTNHIthlBOMh44ia9aPjMlLLXsfJPs/640?wx_fmt=png&from=appmsg "")  
  
第6步：再次触发错误，验证污染是否成功  
  
在已注入   
__proto__ 的请求体中，再次故意破坏   
JSON 语法。  
  
发送，并仔细观察返回的错误对象。  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/qg1MKHx3jGHAgZagrANXKCnUSD4JFK6IuvfNfAlO3dNWMNYQz2a5BffSIbJLmT4ZXHTNL2t37FmRLYomaB8Gms1ru4qyxErQa7UYKVG7NqU/640?wx_fmt=png&from=appmsg "")  
  
如果   
status 和   
statusCode 字段都变成了你设定的   
500，而不是原来的   
400，那就成了。 这证明你成功污染了   
Object.prototype，整个应用里后续读取   
status 的地方都在用你的值。  
  
实操完这六步，这个  
服务端原型污染就算拿下。把这个思路平移到其他功能点，下一个高危赏金就是你的了。
