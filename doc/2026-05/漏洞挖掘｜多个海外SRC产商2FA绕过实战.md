## 0x02 2FA Bypass场景案例  
  
**一：第三方登录绕过**  
  
第一个是比较经典的，是Hackerone里的Web3厂商存在第三方登录绕过2FA。在正常登录流程当中，当我们使用账号密码登录的时候是需要输入OTP。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNEb4KTguwIoic8Yp4PphUNMrNmuoicb361icvr2c9yKAXYLu6BEsok2HLicQ/640?wx_fmt=png&from=appmsg "")  
  
但是这个平台提供了第三方平台登录，我们可以通过Google直接登录。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNE2mSptpq7icy8jLNa1u2ibjtgrmcibrMZEtspveYjQCUeQibN5K4ufaviaNg/640?wx_fmt=png&from=appmsg "")  
  
通过Google账号我们可以顺利登录并且不需要输入One Time Password  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNE0cg7NEvgYwDUMUgME7QtV0DPWZKGLVBibKeNaB8Tlib76ZRv079WY1Bg/640?wx_fmt=png&from=appmsg "")  
  
**二：授权检查缺失**  
  
正常流程下，只有在通过 OTP 验证后，服务器才应该给 Cookie 赋予“已登录”权限。如果系统在输入 OTP 之前就发放了具有完全访问权限的会话 Cookie（或提前激活了权限），攻击者就可以跳过 OTP 界面，直接拿着这个 Cookie 去请求后台数据。  
  
这里是YesWeHack里的支付网关厂商存在的授权检查缺失导致的2FA Bypass。  
  
当我输入账号密码登录的时候，在输入OTP的界面当中，请求里是已经有Auth请求头了。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNEbvelvKaENq3rOiaG9awbubvoziaSPasl7JaSoxEyKeT0q75JWiccM6yJw/640?wx_fmt=png&from=appmsg "")  
  
于是我使用这个还没有经过OTP授权的Auth Token凭证对该账号进行内容的请求，发现在没有经过OTP的Auth Token依旧能够使用，从而导致了2FA Bypass。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNEiare5v7vF7ygJl0NHicf0rGhkO9DUwIUDRz1IWWM3ibpEK0fJF19GQEaw/640?wx_fmt=png&from=appmsg "")  
  
**三：令牌复用**  
  
正常流程，通过OTP获取的令牌应该具备两个特性：  
- 一次性 (One-time)： 一旦被使用（验证成功），后端应立即令其失效。  
  
- 特定绑定 (Context Binding)： 令牌应该只对当前的特定操作（比如那次特定的邀请）有效。 在该案例中，后端没有在令牌使用后将其销毁，导致了重放攻击 (Replay Attack)。  
  
这是Bugcrowd里的某个银行厂商存在的2FA Bypass问题，我们在邀请用户这个功能当中，每次操作都是需要输入OTP。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNEFpHeQbsTibqXscWuqt0YvYRg4KOib35fe0X6q9QmkyPbSJBQZxhE548A/640?wx_fmt=png&from=appmsg "")  
  
这里我们通过正常的一个流程请求将该邮箱成功邀请为Owner账户，并且获取到了令牌Hmac的参数值。  
  
这个平台还有一个机制，就是你如果将这个邮箱添加为User账户需要输入2FA Code，如果你要提升权限将其更改为Owner账户还是需要再次输入新的2FA Code获取新的Hmac的值。  
  
这里我们拿到了令牌后，将这个邮箱邀请为User账户。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNEcFHJR4w0QLuHx2Rb6JSB9qK1ibrkBr4icha6bqChmWQqiaelVkaOXNL2g/640?wx_fmt=png&from=appmsg "")  
  
这里我们通过API，再使用已经被使用过的令牌将这个邮箱添加为Owner账户。发现账户权限成功发生变更，从User变为Owner。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNE0AdxleDzic3sS4NpBL7NSqUm1xUquqJLRKL7TW48XmhsXvAOibh7UiaBg/640?wx_fmt=png&from=appmsg "")  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNEnPDLO2eEWgfojQR1eW4qg51kW8b0zbHPCRP9dibEv0oOzB7ibrPFZOmg/640?wx_fmt=png&from=appmsg "")  
  
**四：OTP重放攻击**  
  
OTP 的设计初衷是   
  
（时间窗口） 和   
  
（单次使用）  
  
如果一个OTP能够重复使用来获取Auth令牌，那么就是OTP重放攻击，Hackerone里的一个电子钱包厂商就存在这类问题。  
  
初步分析成因是缺少使用记录（Used Token Tracking）： 服务器没有记录哪些 OTP 已经被成功使用过。因为OTP在过期一分钟以内可以无限次重复使用，时间偏移上比较宽松。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNEbta4wiaL5gFfpBibu79mTMlnROcLqrfMPjSyaKD8Shks0TQWGt0SpbQw/640?wx_fmt=png&from=appmsg "")  
  
在我们正常登录的流程，该网站会生成LoginToken，为了下一步同步OTP验证需要。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNEZ1icpaADGEHJYKSSYDQhhfMrAYGYhfPNYibr0RU4MPXxr2MibAjy4Y2yw/640?wx_fmt=png&from=appmsg "")  
  
这里我们已知2FA Code：218198，那么我们搭配一开始使用账号密码登录获取到的loginToken进行利用，通过在一分钟之内的多次请求获取到了大量的AuthToken。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNE6SsPTEuO3WmN4xF2qjaCcLD5QQZXGUO44s6HwoAe9AZS4veULjwVqQ/640?wx_fmt=png&from=appmsg "")  
  
这里我们使用重复请求获得的大量AuthToken进行个人信息访问，发现均可成功。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/icefLCXrhxfgoz1D7rLSxNSShiaKFIaMNErmrldMdENkdSEM4xfeA9QK7qEmAbVHCJRmqT0h3icgoQSrRA7ohvOcA/640?wx_fmt=png&from=appmsg "")
